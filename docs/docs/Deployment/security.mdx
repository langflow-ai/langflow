The Langflow UI is an IDE and code execution platform, which means that Langflow is inherently capable of executing arbitrary, developer-provided code.
By design, the Langflow UI includes a code editor allowing developers to author and execute arbitrary Python with full access to the host Langflow backend process, filesystem, and network.
In addition, some components wrap and use code execution, including use of AI models to generate code to be executed based
on user input.

Langflow neither enforces isolation between users within a single Langflow process, nor restricts access to the local disk or network resources. Flow visibility and user access controls are designed for usability, not security enforcement. Users can access underlying database connections and system resources directly. For multi-tenant deployments, Langflow relies on infrastructure-level security rather than application-level isolation, which means it is your responsibility to enforce tenant isolation.

:::warning
You are responsible for the following:

* Enforcing comprehensive infrastructure isolation.
* Running flows safely, particularly flows that use LLM-generated code or user inputs that could contain user-submitted code.
* Ensuring that end-user interactions with Langflow-backed APIs are secure from end-to-end, including data at rest and in transit.
:::

## Secure Langflow during local development

Langflow is a code execution platform with full access to your local system.
You are responsible for ensuring the safety of flows you execute.

In the event that Langflow could execute untrusted or LLM-generated code, consider [blocking custom component execution](./deployment-block-custom-components.mdx) and running Langflow in an isolated and containerized execution environment.
For more information, see [Containerize a Langflow application](/develop-application).

## Secure first-party deployments

When you are serving APIs backed by flows that you or your organization authored, you are responsible for ensuring Langflow-based APIs provide security to your end users.

Follow industry best practices for APIs:

* Use a secure API gateway to provide authentication and authorization
* Ensure user data is appropriately isolated
* Sanitize inputs and outputs against XSS and injection attacks, including regex patterns to prevent ReDoS vulnerabilities

For more information on setting up a reverse proxy, see [Deploy Langflow with Nginx and SSL](/deployment-nginx-ssl).
For more information on authentication configuration, see [API keys and authentication](/api-keys-and-authentication).

## Secure third-party deployments

If you provide Langflow as a service to third parties, you must assume that any code executed by Langflow could be potentially malicious.

Langflow provides no isolation between tenants, so it's your responsibility to enforce isolation at the infrastructure level.

* Use process-level isolation to prevent tenants from sharing a single Langflow process.
* Use disk-level isolation to prevent shared access to writable persistent storage.
* Use network-level isolation to prevent access to private networks.
* Use database-level isolation to prevent access to or modification of shared database resources.

Authentication and authorization must be provided and enforced outside of the Langflow container.
For shared services, such as databases, enforce access limitations externally through credentials and security policies.

For more information, see [Best practices for Langflow on Kubernetes](/deployment-prod-best-practices).

## API key IP restrictions

Langflow supports IP-based access control for API key authentication.
You can restrict which IP addresses are permitted to use a given API key, providing an additional network-level firewall on top of key-based authentication.

There are two levels:

* **Per-key allow-list:** Set directly on each key in **Settings > Langflow API Keys** using the **IP Restriction** column. Accepts semicolon-separated entries. Each entry can be a literal IPv4/IPv6 address (`203.0.113.5`, `2001:db8::1`), a CIDR network (`10.0.0.0/8`, `2001:db8::/32`), or an IPv4 wildcard with `%` for any octet (`10.0.%.%`). Invalid patterns are rejected on save.
* **Global allow-list (`LANGFLOW_API_IP_RESTRICTION`):** A single environment variable that applies to every API-key request across the entire deployment. When set, per-key restrictions are ignored. This is the only available option when `LANGFLOW_API_KEY_SOURCE=env`. Invalid patterns cause Langflow to fail to start, so misconfiguration cannot silently weaken security.

### Extracting the real client IP

`LANGFLOW_TRUST_PROXY_HEADERS` controls how `X-Forwarded-For` and `X-Real-IP` are used:

* `auto` (default) - trust the headers only when the direct TCP peer is a loopback / RFC1918 / ULA address. Safe for both directly-exposed and standard reverse-proxy deployments (Nginx, Apache, K8s ingress, load balancers on a private network).
* `always` - trust the headers unconditionally. Use only when every network path to Langflow passes through a header-sanitizing proxy.
* `never` - ignore the headers and always use the TCP peer.

When headers are trusted, Langflow reads the `LANGFLOW_TRUSTED_PROXY_HOPS`-th entry from the right of `X-Forwarded-For` (default: `1`, i.e. the value appended by the last trusted proxy). Increase this when multiple proxies sit in front of Langflow.

:::danger
Do not use `LANGFLOW_TRUST_PROXY_HEADERS=always` on an instance that is directly reachable from untrusted networks. In that mode a client can set `X-Forwarded-For` to any value and bypass IP restrictions. The default `auto` mode is safe for both topologies and should be preferred.
:::

For full configuration details, see [Restrict API keys by IP address](/api-keys-and-authentication#ip-restriction).
