# Phase 5.5: Component Index Build Caching — Context

## Phase Goal (from ROADMAP)

On cold start, the component index is loaded from the persisted cache instead of rebuilt from a full package walk, bringing the index-populated path under **500ms** vs the **3.4s** rebuild measured on `lfx_with_flow` (2026-04-18 snapshot). Reuses the versioned, atomic on-disk cache that Phase 2 IDX-04/05 already writes.

## Scope Anchors

- **In scope:** read-path short-circuit in `get_and_cache_all_types_dict` using the cache written by Phase 2 IDX-04/05; parity test covering cache-hit vs cache-miss paths.
- **Out of scope:** any change to the write path (already landed in Phase 2); any change to stale-index warning semantics (IDX-07 is the source of truth); platform-specific cache locations (keep the existing `_get_cache_path()`).

## Phase 2 Landing Points (already implemented)

- `src/lfx/src/lfx/interface/components.py:108` — `_read_component_index(custom_path)` returns `dict | None`, validates SHA, returns None on any failure.
- `src/lfx/src/lfx/interface/components.py:216+` — writer stamps `version = version("lfx")` (IDX-04) and writes atomically via tmp + `os.replace` (IDX-05).
- `src/lfx/src/lfx/interface/components.py:685` — `get_and_cache_all_types_dict(settings_service, telemetry_service=None)`. Currently emits the IDX-07 stale warning outside the lock, then acquires the lock and always rebuilds via `import_langflow_components` + `_determine_loading_strategy`.
- `src/lfx/src/lfx/interface/components.py:700` — the IDX-07 peek already loads the cache blob into memory once per cold start; the new read-path should reuse that buffer rather than re-reading the file.

## Locked Decisions (auto-selected defaults)

### D-01: Cache-hit detection
**Decision:** A cache-hit requires three conditions, checked in order: (1) `cache_path.exists()`, (2) `installed_version is not None`, (3) `cached_version == installed_version` AND the blob parses as a non-empty dict with a `types` (or equivalent) payload. Any failure falls through to the full rebuild, no exception raised.
**Why:** Matches the IDX-07 peek logic already in place — reuse the same tri-state (missing / mismatch / OK) so there is a single source of truth for "is this cache usable".
**Source:** ROADMAP success criterion 3.

### D-02: Where to short-circuit
**Decision:** Inside `get_and_cache_all_types_dict`, after the IDX-07 peek block, before the `async with component_cache.lock:` acquisition — but ONLY when the peek block confirmed a fresh, parseable cache. On confirmed hit, assign the cached dict into `component_cache.all_types_dict` under the existing lock (double-check pattern) and return. On miss, proceed to the existing rebuild path.
**Why:** Reuses the IDX-07 peek's already-loaded `cached_blob` so the cache file is read exactly once per cold start. Keeps the lock hold as short as the pre-5.5 implementation.

### D-03: Re-entry safety (IDX-01 interaction)
**Decision:** The short-circuit MUST run inside the lock's double-check, identical to the existing rebuild path. Pattern: peek outside lock, acquire lock, re-check `component_cache.all_types_dict is None`, assign from peek blob, release. If a concurrent task populated via rebuild, the short-circuit is a no-op.
**Why:** The existing `TestIDX01LazyLock::test_cache_built_once_threading` test already guards this invariant; breaking it regresses Phase 2.

### D-04: Parity test shape (IDX-09)
**Decision:** One parity test at `src/lfx/tests/unit/test_component_index_parity.py` (or the existing phase-2 parity module, whichever the pattern-mapper agent picks). It loads a representative flow (`noop_flow.json` plus one flow with ≥3 distinct component types) through both paths and asserts byte-identical `all_types_dict` AND byte-identical vertex execution order via the Phase 2 `_capture_parity_snapshot` helper.
**Why:** ROADMAP success criterion 4 is explicit: "byte-identical final output + vertex execution order". Using the Phase 2 helper keeps the snapshot format consistent.

### D-05: Performance gate (<500ms)
**Decision:** Add an assertion in the parity test (or a sibling perf test) that the cache-hit path measured via `time.perf_counter()` around the `get_and_cache_all_types_dict` call is under 500ms on the same machine. No CI-wide perf gate — `lfx_with_flow` + the new `lfx_with_flow_prebaked` already track this at the benchmark layer.
**Why:** A unit-level perf assertion catches regressions early without waiting for the CI benchmark matrix. Benchmark-layer evidence remains authoritative for the public 500ms claim.

### D-06: Stale / corrupt cache handling
**Decision:** Unchanged from Phase 2 IDX-07. Short-circuit fall-through is silent (no additional log), since the IDX-07 peek already emits the warning for version mismatch, and `_read_component_index` logs for corrupt content. Do NOT delete the bad cache file — the user may want to inspect it.
**Why:** Preserves existing behavior. The phase is a read-path optimization, not a cache-management policy change.

### D-07: Telemetry (optional parameter)
**Decision:** When the cache-hit path is taken, skip the `telemetry_service.log_package_buildtime` call if one exists in the rebuild branch — there is no component-build work to report. Preserve the existing telemetry shape on the miss path.
**Why:** Telemetry should report what actually happened. A "build time of 0ms" metric is noise.

### D-08: Verification step — PR push + CI watch
**Decision:** At the end of execute-phase for Phase 5.5, the orchestrator MUST (1) push the branch to `origin/cold-start-improvements-v2`, (2) invoke `gh run watch <run-id> --exit-status` on the resulting Cold Start Benchmark workflow run, and (3) only mark the phase `passed` once the watched run completes with no new failures vs the prior run. This replaces the manual HUMAN-UAT step used in Phase 5.
**Why:** Phase 5 surfaced three latent bugs only on the CI build. Locking in an automated push-and-watch step makes CI signal part of the phase verification loop instead of a post-hoc human check.
**How to apply:** The gsd-verifier agent should call `git push` and `gh run watch` during its `verify_phase_goal` pass. If the watch exits non-zero due to pre-existing flakes (e.g., `bench:langflow_run_http_ready`), the verifier should report them as "pre-existing, not this phase" rather than blocking.

## Deferred Ideas (not in scope)

- **Cache warming on install.** A post-install hook that pre-renders the index at `uv pip install lfx` time would make the first cold start free. Defer to a future phase.
- **Binary cache format.** `orjson` is already fast, but a pickle or msgpack format would be faster to deserialize for 3-4MB indexes. Measure first, then consider.
- **Incremental cache invalidation.** Rebuild only the changed component subtree on version bump rather than the whole index. Requires change-detection on the component registry; defer.
