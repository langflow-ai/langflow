# Phase 5.5: Component Index Build Caching -- Research

## Summary

Phase 2 (IDX-04/05/07) already put all the pieces on the board: the cache is written
atomically, stamped with `version("lfx")`, and the IDX-07 peek reads the cache blob
outside the lock on every cold start. Phase 5.5 adds a read-path short-circuit: when
the peek confirms a version match, assign the blob directly into `component_cache.all_types_dict`
under the existing lock (double-check pattern) and return, skipping `import_langflow_components`
entirely.

The codebase matches what CONTEXT.md D-01 and D-02 assume in every structural detail.
One nuance needs the planner's attention: the cache blob read by the IDX-07 peek is in
`entries` format (list of `[top_level, components]` pairs), not the flat merged dict that
`component_cache.all_types_dict` expects. The short-circuit must reconstruct the merged
dict from `blob["entries"]`, exactly as `_load_from_index_or_cache` already does at
`components.py:348-354`. The `_build_code_hash_lookups` call must also run on the cache-hit
path so `type_to_current_hash` and `all_known_hashes` are populated.

**Primary recommendation:** Implement IDX-08 inside `get_and_cache_all_types_dict` by
reusing the already-loaded `cached_blob` from the IDX-07 peek block, reconstruct the merged
dict via the same `entries` iteration pattern as `_load_from_index_or_cache`, and call
`_build_code_hash_lookups(component_cache)` before returning. IDX-09 extends the existing
`test_component_index.py` file using `_parity_helpers._reset_component_cache_singleton` and
`_parity_helpers._fake_settings_service`, with `monkeypatch("lfx.interface.components._get_cache_path", ...)`
to control whether a cache file exists.

**Status: VERIFIED -- matches D-01/D-02 exactly.**

`_get_cache_path()`. D-01 and D-02 both say "reuse the IDX-07 peek blob" -- which correctly
targets the user cache. This is consistent throughout the code.

### FA-2: The IDX-07 peek block and blob scoping

**Status: VERIFIED -- blob IS available to reuse; no scope problem. One shape difference
must be handled.**

IDX-07 peek block [components.py:700-748]:
- Runs OUTSIDE `async with component_cache.lock:` (by design; documented at lines 706-719).
- Reads `cache_path = _get_cache_path()` [line 730].
- If `cache_path.exists()`, reads via `orjson.loads(await asyncio.to_thread(cache_path.read_bytes))`
  [line 735] into a local `cached_blob`.
- Extracts `cached_version = cached_blob.get("version")` [line 736].
- Emits warning only on MISMATCH [line 737-744]. Does NOT emit on match.
- Falls through to `async with component_cache.lock:` [line 750] regardless of outcome.

The local `cached_blob` variable is in scope from the `try:` block at line 735. It is
NOT dropped before reaching the lock -- the `except Exception: pass` at line 745 only
handles the corrupt/read-error path. On a clean read with version match, `cached_blob`
survives to line 750.

**Shape difference (PITFALL -- see Pitfalls section):** `cached_blob` is the raw cache
JSON: `{"version": ..., "metadata": ..., "entries": [...], "sha256": ...}`. The `entries`
field is `[[top_level, components], ...]`. `component_cache.all_types_dict` expects
`{top_level: components, ...}` (flat merged dict, no wrapper). The short-circuit must
reconstruct the flat dict from `blob["entries"]`, applying
`filter_disabled_components_from_dict` exactly as `_load_from_index_or_cache` does
[components.py:348-354, 370-376]. Skipping this step produces a dict with wrong shape.

### FA-6: Test isolation patterns for cache-path control

**Status: VERIFIED -- established pattern is `monkeypatch("lfx.interface.components._get_cache_path", ...)`.**

Across `test_component_index.py`, the canonical pattern is:


To simulate cache-hit: write a valid JSON blob to `cache_file` before calling the code

`monkeypatch.setattr(ci, "_get_cache_path", lambda: cache_file)` (module-attribute form)

be called before exercising `get_and_cache_all_types_dict` so a prior test's warm cache does
not satisfy the cache-hit check without reading the file.

## Pitfalls

### P-1: Blob shape mismatch -- must reconstruct flat dict from `entries`

The IDX-07 peek reads the raw cache blob. `component_cache.all_types_dict` must be a flat
`{type_name: template, ...}` dict keyed by component name. The cache `entries` field is
`[[top_level, {component_name: template, ...}], ...]`. The short-circuit must iterate
`entries` and merge into a flat dict, then apply `filter_disabled_components_from_dict`,
exactly as `_load_from_index_or_cache` does at components.py:348-354 and 370-376. Omitting
the Astra filter could expose disabled components in cloud deployments.

### P-2: `_build_code_hash_lookups` must run on the cache-hit path

`type_to_current_hash` and `all_known_hashes` are populated by `_build_code_hash_lookups(component_cache)`
at components.py:769, called at the END of the rebuild branch. If the cache-hit path returns
without calling this function, `component_cache.type_to_current_hash` remains `None` (its
initial value, semantically "not yet loaded / fail-closed" per the comment at components.py:46).
`ensure_component_hash_lookups_loaded` [flow_validation.py:236-252] checks
`component_cache.type_to_current_hash is None` to decide whether to call
`get_and_cache_all_types_dict` again -- a second spurious rebuild. All existing parity tests
exercise `_build_code_hash_lookups` implicitly via the rebuild path; IDX-09 must explicitly
assert `component_cache.type_to_current_hash is not None` after a cache-hit call.

### P-3: `import_langflow_components` has no global-registry side effects to preserve

`import_langflow_components` [components.py:533-567] calls `_load_production_mode` which
calls `_load_from_index_or_cache` or `_load_components_dynamically`. `_load_components_dynamically`
calls `importlib.import_module(modname)` for each component module -- this registers the
module in `sys.modules` as a side effect of Python's import machinery. However, on a
cache-hit path, these imports never happened on the pre-5.5 code either (the shipped
`_assets/component_index.json` path already skips `_load_components_dynamically`). The
production code path is `_load_from_index_or_cache` returning `index_source="builtin"` or
`"cache"`, which does NOT call `importlib.import_module`. No `sys.modules` side effects are
skipped by the short-circuit that were previously present.

### P-4: IDX-07 peek variables are in a `try` block -- the blob is only available on CLEAN READ

The `cached_blob` local only exists when `orjson.loads(await asyncio.to_thread(...))` succeeds.
On a corrupt-file read error, the `except Exception: pass` at line 745 swallows the
exception and execution falls through to the lock. The short-circuit logic must be structured
so that it only activates when `cached_blob` was successfully assigned AND
`cached_version == installed_version`. One clean implementation: refactor the peek block
to also set a `_cache_hit_blob` local (initially `None`) when ALL three D-01 conditions
pass (exists, installed_version not None, versions match, blob parses with valid entries),
then use that in the double-check inside the lock. This keeps the version-mismatch path
(IDX-07 warning emission) unchanged.

### P-5: IDX-01 threading test isolation -- `_get_cache_path` must be patched

`TestIDX01LazyLock::test_cache_built_once_threading` already patches `_get_cache_path` to
a non-existent path (added in 02-06 to prevent the IDX-07 peek from reading the developer's
real ~5.9MB on-disk cache inside ten concurrent threads). After 5.5 lands, the same test
will exercise the new short-circuit code if a real cache exists at the patched path. The
existing patch (pointing at a non-existent file) continues to work correctly -- no cache hit
fires, the fallback path runs, the test passes unchanged. The IDX-09 parity test MUST also
patch `_get_cache_path` so it exercises the exact hit/miss scenario under test.

### Plan IDX-08: Read-path short-circuit in `get_and_cache_all_types_dict`

**File modified:** `src/lfx/src/lfx/interface/components.py`

**Tasks (sequential within the plan):**

**Task 1 -- Promote the IDX-07 peek to return a usable blob**

Inside the existing IDX-07 peek block [components.py:728-748], add a `_pending_cache_hit`
local (initialized to `None` before the peek block). When ALL conditions pass:
- `installed_version is not None`
- `cache_path.exists()`
- `cached_blob` parses as a dict with `"entries"` key and non-empty list
- `cached_version == installed_version`

Set `_pending_cache_hit = cached_blob`. Leave the warning path unchanged (it only fires
on mismatch, not on match).

**Task 2 -- Short-circuit inside the lock**

Inside `async with component_cache.lock:`, BEFORE the double-check rebuilds:

```python
async with component_cache.lock:
    if component_cache.all_types_dict is None:
        if _pending_cache_hit is not None:
            # Cache-hit path: reconstruct flat dict from entries
            merged: dict[str, Any] = {}
            for top_level, components in _pending_cache_hit["entries"]:
                if top_level not in merged:
                    merged[top_level] = {}
                merged[top_level].update(components)
            merged = filter_disabled_components_from_dict(merged)
            component_cache.all_types_dict = merged
            _build_code_hash_lookups(component_cache)
            # D-07: skip telemetry (no build work happened)
        else:
            # Existing rebuild path (unchanged)
            await logger.adebug("Building components cache")
            langflow_components = await import_langflow_components(settings_service, telemetry_service)
            ...
```

`filter_disabled_components_from_dict` is already imported at components.py:20.

**Acceptance criteria:**
- `grep -cE "_pending_cache_hit" src/lfx/src/lfx/interface/components.py` returns >= 2 (set + use)
- `grep -cE "filter_disabled_components_from_dict" src/lfx/src/lfx/interface/components.py` returns >= 2 (already present + new use inside lock)
